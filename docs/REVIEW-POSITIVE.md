# Positive Review — 아키텍처 및 구성의 강점

## 범위

이 문서는 `docs/OVERVIEW.md`, `docs/ARCHITECTURE.md`, `docs/ADR.md`, `docs/PRD.md`, `docs/API.md`, `docs/OPEN-QUESTIONS.md`, `docs/FUTURE-TASKS.md`를 기준으로 현재 설계를 긍정적으로 검토한 결과를 정리한다.

---

## 총평

현재 설계는 "작게 시작하되, 나중에 버릴 가능성이 낮은 경계는 먼저 고정한다"는 원칙이 비교적 잘 지켜져 있다. 특히 MVP 범위를 강하게 제한하면서도, 저장소 교체·언어 확장·검색 고도화 같은 이후 확장 지점을 ADR과 어댑터 경계로 분리해 둔 점이 좋다.

---

## 긍정적 평가

### 1. MVP 범위와 비-목표가 선명하다

- PRD와 ADR이 모두 "정확 semantic resolver가 아닌 경량 코드 탐색 플랫폼"이라는 제품 경계를 명확히 고정하고 있다.
- 호출 그래프, cross-file resolution, 벡터 검색, 그래프 분석, OIDC/RBAC 등을 명시적으로 제외해 과잉 구현 가능성을 낮췄다.
- 이 정도 선명도는 일정과 기대치를 동시에 관리하는 데 유리하다.

### 2. 어댑터 경계가 설계의 중심축으로 잘 잡혀 있다

- `storage/db`, `storage/blob` 분리는 Variant A와 Variant B를 기술적으로 분기시키는 핵심 축인데, 문서상 그 의도가 일관된다.
- 비즈니스 로직이 저장소 구현 세부사항 대신 인터페이스에 의존하도록 설계돼 있어, MVP 이후 확장 비용을 통제하기 좋다.
- 특히 "라우트는 그대로 두고 저장소만 교체"라는 방향은 운영 환경 전환 시 가장 효과적인 단순화 포인트다.

### 3. 업로드와 조회 규약이 결정론적으로 정의돼 있다

- `(repo, commit)` 기준 멱등 업로드, `repo_head` 기반 브랜치 해석, `symbol_key` stable hash 정책은 모두 재현성과 디버깅 가능성을 높인다.
- AI 에이전트, CLI, 웹 UI가 같은 결과를 보게 하려면 이런 규약이 먼저 고정돼야 하는데, 그 부분이 잘 되어 있다.
- 특히 ADR-009처럼 "비결정적 latest 해석 금지"를 명시한 것은 장기적으로 매우 중요한 선택이다.

### 4. 계약 중심 설계가 좋다

- Zod를 REST, MCP, 타입 정의의 단일 출처로 두려는 방향은 유지보수성과 계약 일관성 측면에서 타당하다.
- OpenAPI 자동 생성까지 이어지는 구조는 문서와 구현의 동기화 비용을 줄인다.
- CLI가 내부 DB를 직접 건드리지 않고 HTTP 계약을 따르도록 한 결정도 시스템 경계를 지키는 데 도움이 된다.

### 5. 기술 선택이 목표와 비교적 정합적이다

- Node.js 22 + TypeScript + Hono + pnpm workspace 조합은 MCP, CLI, 서버를 한 런타임으로 묶기 좋다.
- `better-sqlite3`, zip, docker-compose 같은 선택도 "3~10명 팀이 별도 SRE 없이 운영"이라는 목표와 맞는다.
- 즉, 최신성보다 운영 단순성과 구현 속도를 우선한 선택이 설계 목표와 충돌하지 않는다.

### 6. 읽기 전용 MCP 전략이 현실적이다

- 중앙 인덱스 서버에 대해 MCP를 읽기 전용으로 제한한 점은 권한·보안·운영 부담을 동시에 줄인다.
- AI 에이전트가 탐색만 담당하고 실제 쓰기 권한은 다른 시스템에 남기는 구조는 MVP에서 안전한 기본값이다.
- tool surface를 4개로 제한한 것도 도구 설계 복잡도를 적절히 제어한다.

### 7. 압축 포맷 선택의 근거가 분명하다

- zip 채택 사유가 "압축률"이 아니라 "단일 파일 랜덤 액세스"라는 서비스 요구와 직접 연결돼 있다.
- `/body` 조회를 빠르게 만들기 위해 저장 포맷을 역으로 선택한 점은 기능 중심의 좋은 설계다.

### 8. 확장 로드맵이 무분별하지 않다

- FUTURE-TASKS가 단순 아이디어 목록이 아니라, 트리거 조건과 예상 공수까지 붙어 있다.
- "실측 병목 없이는 도입하지 않는다"는 원칙이 ADR과 Future Tasks 모두에 반복된다.
- 이 방식은 기술 부채를 만들지 않으면서도 팀 내 논의를 생산적으로 유지하게 한다.

### 9. 문서 체계가 비교적 잘 구조화돼 있다

- 개요, 아키텍처, ADR, API, 시작 가이드, 오픈 질문, 미래 과제가 역할별로 나뉘어 있다.
- 리뷰어가 전체 구조를 파악한 뒤 상세 결정을 추적하기 쉬운 편이다.
- 특히 ADR이 단순 결론이 아니라 트레이드오프까지 기록하고 있어 이후 의사결정 복기에 유용하다.

---

## 유지 권장 사항

- ADR-003, ADR-009, ADR-010, ADR-011은 현재 설계에서 가장 가치가 큰 결정이다. 구현 과정에서 이 경계가 흐려지지 않도록 유지하는 것이 좋다.
- 문서가 약속한 "계약 일관성"과 "결정론적 해석"은 이 시스템의 핵심 신뢰 기반이므로, 이후 기능 추가보다 먼저 보호해야 한다.
