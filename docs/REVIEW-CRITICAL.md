# Critical Review — 아키텍처 및 구성의 우려사항

## 범위

이 문서는 `docs/OVERVIEW.md`, `docs/ARCHITECTURE.md`, `docs/ADR.md`, `docs/PRD.md`, `docs/API.md`, `docs/OPEN-QUESTIONS.md`, `docs/FUTURE-TASKS.md`와 현재 저장소 구조를 함께 대조해 비판적으로 검토한 결과를 정리한다.

---

## 총평

설계 방향은 실용적이지만, 일부 문서는 "현재 구현", "근시일 내 구현", "장기 확장 구상"이 한 문맥에서 섞여 있어 독자가 실제 시스템의 완성도를 과대평가할 수 있다. 지금 필요한 보완은 새 기능 추가보다, 설계 약속과 실제 구현 상태를 더 엄격하게 구분하는 일이다.

---

## 비판적 평가

### 1. 문서와 실제 구조 사이에 이미 불일치가 있다

- `docs/ARCHITECTURE.md`는 `packages/server/src/mcp/`, `services/`, `migrations/`, `packages/cli/src/extractors/python.ts`, `commands/status` 등을 현재 구조처럼 서술하지만, 실제 저장소에는 일부가 아직 없다.
- `docker/`, `infra/`, `scripts/`도 문서의 주요 디렉토리로 등장하지만 현재 워크트리 기준으로는 설계 문맥이 더 앞서가 있다.
- 이 상태가 지속되면 문서가 "설계서"가 아니라 "희망 구조도"로 읽히기 쉽다.

### 2. Variant B 지원은 아직 설계상 동치이지, 구현상 동치가 아니다

- ADR은 A/B를 동일 API와 동일 비즈니스 로직으로 설명하지만, 현재 B는 skeleton 단계이고 핵심 저장소 구현이 비어 있다.
- 마이그레이션 전략도 SQLite는 구체적이지만 PostgreSQL은 추후 결정이다.
- 따라서 지금 문서 표현은 "교체 가능"보다 "교체 예정"에 가깝고, 이 차이를 더 명확히 써야 한다.

### 3. 성능 목표 대비 병목 분석이 아직 낙관적이다

- 단일 프로세스, `better-sqlite3` 동기 호출, zip 기반 본문 추출, 업로드와 조회 공존이라는 조합은 단순하지만 경합 지점이 명확하다.
- ADR도 이를 인정하지만, p95 검색 300ms와 업로드 60s 목표가 어떤 데이터 분포와 동시성 조건에서 성립하는지 문서에 실측 근거가 없다.
- 특히 `adm-zip`의 메모리 사용량, FTS5 트리거 유지 비용, 대형 업로드 중 read latency 저하는 아직 설계상 위험으로 남아 있다.

### 4. 보안 모델은 MVP 기준으로도 설명이 더 엄격해야 한다

- 쓰기는 단일 정적 Bearer token, 읽기는 무인증이라는 모델인데, 이는 "사내망/VPN 전제"가 깨지는 순간 바로 취약해진다.
- 토큰 회전, 감사 로그, 사용자 단위 식별, 레포별 권한 분리가 모두 없다.
- 설계서가 이 모델을 "MVP에서 허용"이라고 쓰는 것은 괜찮지만, 운영 경계와 금지 시나리오를 더 강하게 명시해야 한다.

### 5. 정확도 한계가 AI 사용 시 오해를 만들 수 있다

- 문서는 tree-sitter의 한계를 솔직하게 인정하지만, 실제 사용자는 `references`나 검색 결과를 "정확한 의미 해석"으로 오인할 가능성이 높다.
- 특히 이름 기반 occurrence 매칭은 동명 심볼이 많은 코드베이스에서 거짓 양성(false positive)을 쉽게 만든다.
- AI 에이전트가 이 결과를 근거로 설명하거나 수정 작업에 들어가면, 잘못된 컨텍스트를 확신 있게 소비할 위험이 있다.

### 6. 문서 간 API 서술이 완전히 한 목소리는 아니다

- PRD에는 `analyze . -> analyze push -> PUT /v1/indexes` 같은 축약 표현이 나오지만, API 문서는 실제로 `POST /v1/repos/:name/indexes` 이후 다단계 업로드를 정의한다.
- 이런 차이는 작은 수준이지만, 구현자나 리뷰어가 빠르게 읽을 때 잘못된 mental model을 만들 수 있다.
- 핵심 업로드 플로우는 모든 문서에서 동일한 용어와 단계로 맞추는 편이 좋다.

### 7. Detached HEAD와 branch-less 업로드의 UX가 거칠다

- ADR-009는 branch가 없으면 commit-less 조회가 실패할 수 있음을 인정한다.
- 하지만 Getting Started와 제품 흐름은 이를 상대적으로 부드럽게 다루고 있어, 사용자가 "push는 성공했는데 search는 왜 안 되지?" 상황을 맞을 여지가 있다.
- 이 동작은 설계상 합리적일 수 있지만, UX 차원에서는 더 노출돼야 한다.

### 8. 테스트 전략이 아직 설계 신뢰도를 충분히 뒷받침하지 못한다

- 현재 문서 기준 테스트는 smoke 중심이며, 계약 테스트, 마이그레이션 회귀 테스트, 멱등 업로드 복구 테스트, 성능 기준선 테스트가 약하다.
- 이 시스템은 "정확한 해석"보다 "같은 입력에 같은 결과"가 더 중요한데, 바로 그 결정론을 검증하는 테스트 항목이 문서상 충분히 강조되지 않는다.
- 특히 ADR-009, ADR-010, ADR-012는 테스트 없이는 쉽게 깨질 수 있는 규약이다.

### 9. 문서 체계는 좋지만 상태 표기가 부족하다

- 지금 문서 묶음에는 각 항목이 `planned`, `implemented`, `stub`, `deferred` 중 어디에 속하는지 한눈에 보여주는 표식이 거의 없다.
- 그 결과 ADR 완료와 실제 구현 완료가 동일하게 읽힐 수 있다.
- 설계 문서가 많아질수록 이 상태 표기 부재는 오히려 이해 비용을 높인다.

---

## 우선 보완 권장 사항

### 1. 문서 상태를 명시적으로 구분

- 각 문서 또는 섹션에 `현재 구현`, `구현 예정`, `스텁 존재`, `장기 확장` 표기를 추가하는 것이 가장 시급하다.

### 2. 아키텍처 문서를 실제 코드 기준으로 다시 정렬

- 현재 저장소에 없는 디렉토리와 모듈은 "목표 구조"로 분리하고, 실제 구조와 섞어 쓰지 않는 편이 좋다.

### 3. 운영 제약을 더 강하게 문서화

- 읽기 무인증, 정적 토큰, branch 없는 업로드, 이름 기반 reference 정확도 한계를 Getting Started와 API 문서에도 더 전면 배치할 필요가 있다.

### 4. 핵심 규약에 대한 검증 계획 추가

- 멱등 업로드, repo_head 해석, FTS 일관성, zip 본문 조회 정확성에 대한 테스트/벤치 계획이 별도 문서 또는 ADR 후속 항목으로 필요하다.
