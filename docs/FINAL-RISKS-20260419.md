# Final Risks — 2026-04-19

## 목적

이 문서는 현재까지의 코드 리뷰와 수정 반영 이후에도 남아 있는 리스크를 팀 검토용으로 정리한다.  
목표는 "당장 수정해야 할 버그"보다, 다음 설계/우선순위 논의가 필요한 지점을 빠르게 합의하는 것이다.

---

## 전제

다음 항목은 이미 이번 세션에서 보완되었다.

- 업로드 대상 index와 payload 불일치 허용 문제
- finalize lifecycle 무결성 부재
- refs-only 파일 누락
- duplicate `symbol_key` 이후 parent 해석 붕괴
- 동명 caller의 이름 문자열 기반 `caller_key` 오매핑 일부

아래는 그 이후에도 남아 있는 리스크다.

---

## 1. `dependencies`는 여전히 저장/전달되지 않는다

현 상태에서 extractor는 `dependencies`를 만들지만, `PackedIndex`와 서버 저장 경로에는 포함되지 않는다.  
즉 import, extends, implements 정보는 내부 계산까지만 존재하고 최종 인덱스에는 남지 않는다.

영향:

- 파일 간 관계 분석을 later feature로 쓰려면 다시 스키마를 넓혀야 한다.
- 문서에서 관계 데이터가 있는 것처럼 읽히면 실제 기능 기대와 어긋날 수 있다.

논의 포인트:

- `dependencies`를 MVP+1 범위로 올릴지
- 아니면 extractor 내부 산출물로만 남겨두고 문서에서 명확히 제외할지

---

## 2. reference 정확도는 여전히 semantic 수준이 아니다

`caller_key` 안정성은 개선됐지만, `callee_name`은 여전히 이름 기반이다.  
즉 "누가 누구를 호출했는가"는 semantic resolver가 아니라 tree-sitter 기반 후보 매칭 수준이다.

영향:

- 동명 함수/메서드가 많은 코드베이스에서는 false positive가 남는다.
- AI 에이전트가 결과를 정확한 호출 그래프로 오해하면 잘못된 컨텍스트를 소비할 수 있다.

논의 포인트:

- 이 한계를 API/MCP/UI에 더 직접적으로 노출할지
- 향후 SCIP/LSIF 수용 같은 확장 경로를 열어둘지

---

## 3. 검색 계약은 아직 보수적이고 제한적이다

현재 `/v1/search`는 이름 중심 FTS에 가깝고, query 문자도 제한적이다.  
이 제약은 안전하지만 실제 사용성은 낮출 수 있다.

영향:

- 파일 경로, 복합 쿼리, 부분적인 개발자 검색 습관을 흡수하지 못한다.
- MCP/웹 UI에서 "왜 이 검색은 안 되지?"가 반복될 수 있다.

논의 포인트:

- 현재 제약을 유지하고 문서/UX로 설명할지
- 점진적으로 query surface를 넓힐지

---

## 4. Variant B는 여전히 설계 파리티 수준이다

PostgreSQL/S3 경로는 인터페이스와 스텁만 있고, 실제 동작 검증은 없다.

영향:

- 문서가 앞서가면 팀이 "이미 전환 가능하다"고 오해할 수 있다.
- A 편향이 코드에 다시 스며들어도 지금은 자동으로 걸러지지 않는다.

논의 포인트:

- Variant B를 당분간 명시적으로 비활성 범위로 둘지
- 최소한 adapter contract test라도 먼저 둘지

---

## 5. 테스트는 강화됐지만 여전히 smoke 중심이다

이번에 회귀 테스트를 추가했지만, 여전히 대부분은 in-process smoke test다.  
대용량 입력, 성능 기준선, 손상된 zip, 예외적인 parser edge case는 충분히 다루지 않는다.

영향:

- 구조적 버그는 줄었지만 운영 부하나 비정상 입력에 대한 신뢰도는 아직 제한적이다.
- 리팩터링 시 contract regression을 더 넓게 잡아주지는 못한다.

논의 포인트:

- 다음 테스트 투자 우선순위를 성능/계약/fixture 기반 중 어디에 둘지
- CLI와 server의 fixture 세트를 공용화할지

---

## 6. 문서와 실제 구현의 정합성은 계속 관리가 필요하다

앞서 별도 문서로 정리했듯이, 일부 아키텍처 문서는 현재 구현과 목표 구조가 섞여 읽힌다.  
코드가 빠르게 바뀌는 동안 이 차이가 다시 커질 가능성이 있다.

영향:

- 팀원이 문서를 보고 구현 상태를 과대평가할 수 있다.
- 코드 리뷰에서 이미 해결된 문제와 아직 남은 문제의 경계가 흐려진다.

논의 포인트:

- `implemented / stub / planned` 표기를 문서 체계에 넣을지
- `ARCHITECTURE.md`를 실제 코드 기준으로 재정리할지
